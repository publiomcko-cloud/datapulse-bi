# Technical Specification — DataPulse BI

## 1. Overview

DataPulse BI is a full-stack data product focused on lightweight data engineering and business intelligence. The system ingests raw data, stores it in a relational database, transforms it into analytical structures, exposes business metrics through an API, and presents insights in a web dashboard.

The project is intended to function as a realistic portfolio project, showing both software engineering and data engineering ability.

## 2. Project Objective

The objective is to build a complete, reproducible, and deployable data product that demonstrates:

- data ingestion from a structured source
- data validation and cleaning
- relational modeling for analytics
- backend API design
- business metric calculation
- dashboard construction
- local and remote deployment practices
- documentation suitable for recruiters, clients, and technical reviewers

## 3. Product Context

The project simulates a company that currently depends on manual spreadsheets to understand business performance. The system replaces this fragile workflow with a structured data pipeline and dashboard.

The first version should use a simple but realistic dataset, preferably synthetic sales data or a public dataset with business meaning.

## 5. Core Use Case

A business user opens the dashboard and can quickly answer:

- How much revenue was generated in the selected period?
- How many orders were completed?
- What is the average order value?
- Which products or categories performed best?
- Which regions or channels generated more revenue?
- How are indicators changing over time?
- Are there data quality issues in the imported dataset?

## 6. Functional Requirements

### FR-001 — Data ingestion

The system must import data from at least one source.

Initial accepted sources:

- local CSV file
- public API
- synthetic dataset generated by script

The ingestion process must store records in a raw data table before transformation.

### FR-002 — Data validation

The system must validate basic rules before accepting records into analytical tables.

Examples:

- required fields must not be null
- dates must be valid
- monetary values must be numeric
- quantities must be positive
- duplicated source IDs must be handled

### FR-003 — Data transformation

The system must transform raw data into clean analytical tables or views.

Transformations may include:

- date normalization
- currency normalization
- category mapping
- customer segmentation
- revenue calculation
- aggregation by day, product, region, and channel

### FR-004 — Data storage

The system must use PostgreSQL as the main database.

It must separate:

- ingestion logs
- raw records
- transformed records
- analytical dimensions and facts

### FR-005 — Metrics API

The backend must expose endpoints for dashboard metrics.

Minimum endpoints:

- healthcheck
- ingestion status
- KPI summary
- revenue over time
- top products
- revenue by region
- revenue by channel
- orders table with filters

### FR-006 — Dashboard

The frontend must display:

- KPI cards
- trend charts
- ranking charts
- filters
- table with recent records
- ingestion status
- data quality summary

### FR-007 — Filtering

The dashboard must allow filtering by:

- date range
- product category
- region
- sales channel

### FR-008 — Testing

The system must include minimum tests for:

- data validation
- transformation logic
- API healthcheck
- metric endpoint correctness

### FR-009 — Documentation

The repository must include documentation for setup, architecture, database modeling, scope, backlog, testing, deployment, and agent instructions.

## 7. Non-Functional Requirements

### NFR-001 — Reproducibility

A developer must be able to run the project locally by following the setup guide.

### NFR-002 — Clarity

The code and documentation must be clear enough for a recruiter, client, or agent to understand the system.

### NFR-003 — Maintainability

The project must use modular structure and avoid mixing ingestion, transformation, API, and frontend responsibilities.

### NFR-004 — Performance

The MVP does not need big-data scale, but the database and API should handle at least 50,000 sample records without becoming unusable.

### NFR-005 — Observability

The ingestion and transformation process must produce logs with:

- start time
- end time
- status
- processed records
- rejected records
- error message when applicable

### NFR-006 — Security

The MVP may be public and unauthenticated, but it must avoid exposing secrets, credentials, or private data.

### NFR-007 — Deployment readiness

The project must be prepared for deployment with environment variables and documented deployment steps.

## 8. Main Modules

### 8.1 Data ingestion module

Responsible for collecting source data and storing it in raw tables.

### 8.2 Data validation module

Responsible for checking required fields, types, duplicates, and basic consistency.

### 8.3 Transformation module

Responsible for creating clean analytical data structures from raw data.

### 8.4 Database module

Responsible for migrations, schema organization, relationships, constraints, and indexes.

### 8.5 Backend API module

Responsible for serving data to the frontend through clean endpoints.

### 8.6 Frontend dashboard module

Responsible for presenting metrics and insights visually.

### 8.7 Testing module

Responsible for validating business logic and critical flows.

### 8.8 Deployment module

Responsible for publishing the application and keeping configuration manageable.

### Raw layer

Stores source records with minimal modification.

### Staging layer

Stores cleaned and normalized data.

### Analytics layer

Stores fact and dimension tables or materialized views used by the dashboard.
